githug的简单介绍

这是一个通过游戏的形式将开发中使用 git 工具可能碰见的场景转化成一个个的小关卡，闯关者通过一个个关卡的同时，潜移默化地掌握了git知识。

闯过这 54 关，点亮你的 Git 技能树
Github地址

溯本追源–git

时间： 2005 年 4 月
本质：一个分布式 (distributed) 的性能优异的版本控制系统 (VCS) 工具

githug游戏

安装：githug 的安装在最上面给出的的两个相关链接里都有提到，就不重复了。

玩法：每一关的提交命令githug play ，实际上直接输入githug就可以了，查看所有关卡命令githug levels，提示命令githug hint ，关卡重置命令githug reset，跳转到任意关卡命令githug reset 关卡名称，例如：githug reset add，就是跳转到第三关 add

create githug directory
首先创建 githug 目录
yn–>y

1. init

在刚刚创建的 githug 目录下初始化一个 git 仓库
先切换到 githug 目录下cd git_hug，然后 git init

2. config

设置 name 和 email
git config --global user.name "drops"
git config --global user.email "wei..."

3. add

将一个叫 README 的文件，从工作区添加到暂存区 (stageing area) ,以便追踪
git add README

4. commit

提交commit信息
git commit -m "add README"

5. clone

从一个 github 地址克隆仓库
git clone https://github.com/Gazler/cloneme
注：如果仓库地址后面不加参数，则默认 clone 到当前目录，与后面一关对照

6. clone_to_folder

从 github 地址克隆仓库到指定文件夹下
git clone https://github.com/Gazler/cloneme my_cloned_repo

7. ignore

在git仓库中忽略指定文件，即不追踪这些文件
vi .gitignore，打开 vim 编辑 .gitignore 文件

添加这一行 * .swp

8. include

git仓库中停止追踪 .a 结尾的文件，但不包括 lib.a
vi .gitignore，打开 vim 编辑 .gitignore ** ![include](http://7fva1q.com1.z0.glb.clouddn.com/githug%238-include.png) 添加这两行 ** .a**，**!lib.a`*

9. status

查看仓库中未追踪的文件
git status，可以查看待提交的变更和未追踪的文件

10. number_files_commited

查看待提交的文件数目
仍然是 git status`` ，可以直接看到要提交的文件

11.rm

删除一个已经在工作区中删除，但还存在于仓库中的文件
先 git status ，查看仓库状态，发现有一个待删除的文件
git rm 待删除文件 ，删除仓库中的该文件，接着git commit ，将改动提交到仓库中

12.rm_cached

删除已经添加到暂存区的某一文件，但在文件系统中保留该文件
git rm --cached 待删除文件

13.stash

暂存工作状态，但是不提交
git stash

14. rename

修改仓库中某个文件的名称
git mv oldfile.txt newfile.txt，此时修改已经被暂存但还未提交

15. restructure

新建一个文件夹，并将原仓库中的某些文件移到这个新建的文件夹中
git mkdir src，git mv *.html src
其中 git mkdir 目录名 是新建目录， git mv 文件 目录 是移动文件命令

16. log

查看提交日志
git log

17. tag

为最近的一次 commit 打上标签
git tag 标签名

18. push_tags

把一些标签推送到远程仓库
git push tags

19. commit_amend

修改上一次的提交，加入一个新的文件
git add forgotten_file.rb ，先将文件存入暂存区，然后 git commit amend ，使用一次新的提交，替代上一次的提交

20. commit_in_future

为一次提交指定一个未来的时间
git commit --date=12.06.2016T15:00:00

21. reset

把某个文件撤出暂存区
git reset HEAD 要撤出的文件
git reset HEAD to_commit_second.rb

22. reset_soft

只撤销上次的提交信息，工作区和暂存区内容不变
git reset --soft HEAD^1

23. checkout_file

丢弃工作区某个文件的改动，用上次提交覆盖工作区
git checkout [commit] [file]
git checkou con*
注：图中的这个实际上是用暂存区的文件覆盖，实际上 checkout 还有几种，以及需要和 git reset 命令进行区分，这些留待以后说

24. remote

查看远程仓库
git remote

25. remote_url

查看远程仓库以及地址
git remote -v

26. pull

从（远程）仓库取回变动，并与本地分支合并
git pull [remote] [branch]
git pull origin master

27. remote_add

添加一个远程仓库，同时命名，git remote add [shortname] [url]
git remote add origin https://githug.com/githug/githug

28. push

推送本地修改到远程仓库中，这个例子里，远程的仓库和分支是 origin/master ，本地主机的分支是master，先 git rebase origin/master ，在当前分支上，合并 origin/master ，然后通过 git push origin master 推送本地修改到这个远程仓库

git push [remote] [branch]，上传本地指定分支到远程仓库

29. diff

显示暂存区和工作区的差异
git diff，注：diff信息读取也值得一讲，也留待以后细说

30. Blame

显示指定文件是什么人在什么时间修改过
git blame [file]

31. branch

创建分支
git branch [branch_name]

32. checkout

创建并切换分支
git branch my_branch 先创建一个名为 my_branch 的分支，接着 git checkout my_branch ，通过 git checkout 命令切换到该分支
实际上，开发中一般直接使用git checkout -b [branch_name]，直接创建分支并且切换过去

33. checkout_tag

切换到指定 tag
git checkout tag_name，可以先git tag 查看所有的tag ，发现有 v1.2，然后git checkout v1.2，切换过去

34. checkout_tags_over_branch

切换到指定 tag ，但这个 tag 和 branch 重名了
git checkout tags/tag_name，还是先git tag 查看所有的tag，接着git checkout tags/v1.2直接切换

35. branch_at

根据一个特定提交创建一个分支
该关卡的要求是在上一次的提交前创建新分支，git branch test_branch HEAD^1

36. delete_branch

删除某个分支
git branch -d [branch_name]

37. push_branch

推送指定分支到远程仓库中
git push <远程主机名> <本地分支名>:<远程分支名>
该关卡中远程主机名为 origin 因为不想合并到主分支中，直接推送到 test_branch 中 ，
所以git push origin test_branch:test_branch

38. merge

合并指定分支内容到当前分支
git merge [branch_name]

39. fetch

获取远程仓库的修改，但是不合并到当前分支
git fetch [远程主机名]

40. rebase

这个命令，我还不怎么理解，本关卡中，有两个分支 master 和 feature 。
先查看 master 的提交记录

然后查看 feature 的提交记录

接着git rebase master feature
再查看 master 分支的提交记录

似乎是将 feature 分支的提交记录合并了

41. repack

将仓库中冗余的打包文件移除
git rebase

查看Git 是如何存储对象的

42. cherry-pick

cherry-pick命令”复制”一个提交节点并在当前分支做一次完全一样的新提交
git checkout-pick [某个commit的hash值]

在本关卡中，先查看所有的提交记录 git log -all

记录 Filled in REASME.md with proper input 这次 commit 的 hash 值
接着 git checkout-pick ca3236

43. grep

grep 在 linux 中就是一个强大的搜索命令，支持文本、正则等方式
本关卡中git grep TODO

44. rename_commit

重命名某次提交
先git log，查看提交信息，记录有输入错误的 commit 的前一次 commit 的 hash 值
然后git rebase -i hash值
接着会出现两次默认编辑器窗口，第一次是修改 .git/rebase-merge/git-rebase-todo ，将出错的那行 pick 改成 reword ，同时修改后面的错误，保存之后
又会出来一个窗口，这次修改的是 .git/COMMIT_EDITMSG 手动修改提交信息中的错误，再保存就可以了
